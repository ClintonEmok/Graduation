<objective>
Clarify timeline UX semantics on `/timeslicing` and `/timeslicing-algos` for QA/exploration users by making reference-context behavior, selection-focused detail behavior, and route-specific roles explicit.

Purpose: Both routes are QA/exploration surfaces now, but their current UI leaves too much implicit: the detail histogram changes with selection, fetched-domain chips and panel labels are easy to misread, and `/timeslicing` vs `/timeslicing-algos` roles are not stated clearly enough. This plan codifies a single source-of-truth labeling model, applies it to both routes, and preserves a clean path to later general-user adaptation without touching dashboard UX now.
Output: A shared QA timeline labeling model, reusable QA context UI, route rewires for `/timeslicing` and `/timeslicing-algos`, regression coverage, and a manual verification pass.
</objective>

This UX is correct for QA/exploration users now because those users need unambiguous behavioral semantics more than polished consumer simplification. Explicitly naming the stable reference context, the fetched window, and the selection-driven detail view makes it easier to reason about whether an interaction changed data scope, render scope, or just focus. Keeping `/timeslicing` richer and `/timeslicing-algos` diagnostic-heavy matches how these routes are already used in practice.
</qa_rationale>
