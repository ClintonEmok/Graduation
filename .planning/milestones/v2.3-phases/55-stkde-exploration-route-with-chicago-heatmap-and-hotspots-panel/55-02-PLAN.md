<architecture_decision>
### Goal
Support QA-only full-population STKDE on `/stkde` up to the full dataset while avoiding naive row materialization in server JS memory.

### Options considered
1. **Naive full materialization in JS (REJECTED)**
   - Flow: remove 50k cap -> `queryCrimesInRange` returns all rows -> current JS `computeStkdeFromCrimes` processes full array.
   - Pros: smallest code change.
   - Cons: memory spikes, GC churn, unstable latency, high OOM risk for ~8.4M rows, violates anti-pattern requirement.

2. **SQL/chunked aggregation pipeline (CHOSEN)**
   - Flow: in full-population mode, push down aggregation to DuckDB (`GROUP BY` spatial cell + temporal bucket), fetch aggregated chunks/pages, then run deterministic hotspot projection from aggregated counts/windows.
   - Pros: avoids row-level JS materialization, bounded memory, scales to full dataset, keeps deterministic output path.
   - Cons: more implementation complexity; requires aggregation contract and additional tests.

3. **Background job/offline precompute only (NOT CHOSEN for this phase)**
   - Pros: stable runtime for very large requests.
   - Cons: async job orchestration/UI polling is out of scope; slower iteration for QA on-demand tuning.

### Decision
Implement option 2: **SQL/chunked aggregation** for `computeMode=full-population`, with strict safety guardrails and automatic fallback to sampled mode when guardrails trigger.
</architecture_decision>

<api_contract_additions>
- Add request field: `computeMode?: 'sampled' | 'full-population'` (default remains `'sampled'` for backward compatibility).
- Add response provenance fields:
  - `meta.requestedComputeMode`
  - `meta.effectiveComputeMode`
  - `meta.fallbackApplied` (existing, now mode-aware)
  - `meta.clampsApplied` (normalized list)
  - `meta.fullPopulationStats?: { scannedRows: number; aggregatedCells: number; queryMs: number }`
- Guardrails in request/normalization:
  - full-pop mode only accepted when route is `/stkde` caller intent and QA flag is on.
  - explicit max compute window for full-pop requests (for example max epoch span in days); exceed -> fallback sampled + provenance reason.
  - payload cap maintained (`STKDE_RESPONSE_SIZE_LIMIT_BYTES`) with truncation metadata.
</api_contract_additions>

<safety_controls>
- API-side timeout budget for full-pop path (Promise.race deadline, e.g. 20s) -> fallback sampled on timeout with reason `full-pop-timeout`.
- Newest-request-wins + AbortController on `/stkde` remains mandatory.
- Hard cap on returned hotspot rows/heatmap cells remains enforced in both modes.
- Fallback rules (deterministic order):
  1. invalid full-pop request or QA gate off -> sampled
  2. compute window exceeds cap -> sampled
  3. full-pop aggregation timeout/error/resource guard -> sampled
  4. sampled path failure -> 500 (no silent empty success)
- All fallback/clamp events must be surfaced in response `meta` and rendered in `/stkde` UI.
</safety_controls>
