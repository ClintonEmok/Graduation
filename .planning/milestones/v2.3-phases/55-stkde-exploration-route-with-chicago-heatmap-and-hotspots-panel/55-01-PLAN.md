<architecture>
- **Route isolation:** New `src/app/stkde/*` route shell; no imports from suggestion/full-auto workflows.
- **Compute strategy (hybrid):**
  1. **Server (authoritative):** `POST /api/stkde/hotspots` performs STKDE kernel accumulation on bounded data and returns heatmap grid + top hotspots.
  2. **Worker (client refinement):** `stkdeHotspot.worker.ts` handles client-side sorting/filter/projection of returned hotspots for responsive map/list interaction without recomputing kernels in UI thread.
- **State ownership:** `useStkdeStore` owns route-local selection and filters (`selectedHotspotId`, spatial bbox filter, temporal window filter), keeping map/panel synchronization deterministic.
- **Map rendering:** Prefer MapLibre native heatmap layer semantics for grid/intensity rendering (route-local map layer component), with explicit zoom/radius/intensity controls and bounded source size.
- **Non-goal lock:** No dashboard integration; no route-level changes in `/timeslicing` or `/timeslicing-algos` beyond shared utility reuse.
</architecture>

<data_contracts>
- **Request (`StkdeRequest`):**
  - `domain: { startEpochSec: number; endEpochSec: number }`
  - `filters: { crimeTypes?: string[]; bbox?: [minLng, minLat, maxLng, maxLat] }`
  - `params: { spatialBandwidthMeters: number; temporalBandwidthHours: number; gridCellMeters: number; topK: number; minSupport: number; timeWindowHours: number }`
  - `limits: { maxEvents: number; maxGridCells: number }`
- **Response (`StkdeResponse`):**
  - `meta: { eventCount: number; computeMs: number; truncated: boolean; fallbackApplied?: string | null }`
  - `heatmap: { cells: Array<{ lng: number; lat: number; intensity: number; support: number }>; maxIntensity: number }`
  - `hotspots: Array<{ id: string; centroidLng: number; centroidLat: number; intensityScore: number; supportCount: number; peakStartEpochSec: number; peakEndEpochSec: number; radiusMeters: number }>`
  - `contracts: { scoreVersion: 'stkde-v1' }`
- **Determinism rule:** For identical request payload (including ordered params + domain + filters), top hotspot IDs and scores are stable within floating-point tolerance.
</data_contracts>

<performance_constraints>
- API p95 compute latency target: **<= 1500ms** for up to **50k events** and **<= 12k grid cells** on local QA hardware.
- Client interaction latency target (map click <-> list highlight roundtrip): **<= 120ms**.
- Response payload guard: **<= 2.5MB JSON** (truncate/summarize beyond this with explicit `meta.truncated=true`).
- Browser memory ceiling for `/stkde` route working set (excluding base app): **<= 250MB** sustained during repeated parameter sweeps.
- Concurrent request policy: newest request wins; prior in-flight responses are ignored by request ID.
</performance_constraints>

<risks>
- **Compute cost risk:** STKDE kernel accumulation can exceed latency budgets for large ranges.
  - Mitigation: hard caps (`maxEvents`, `maxGridCells`), truncation metadata, newest-request-wins cancellation.
- **Map rendering performance risk:** large heatmap cell payloads can degrade pan/zoom FPS.
  - Mitigation: bounded cell count, zoom-aware radius/intensity, source-size guardrails.
- **Memory pressure risk:** dense response + duplicated client transforms can increase heap usage.
  - Mitigation: server truncation, worker-only lightweight transforms, avoid storing duplicate arrays in multiple stores.
</risks>

<rollback>
- Primary rollback: set `stkdeRoute` feature flag false to disable route surface immediately.
- Secondary rollback: keep route enabled but force coarse preset (`larger gridCellMeters`, lower `topK`) through server-side clamps.
- Emergency fallback: short-circuit `/api/stkde/hotspots` with 503 + explanatory payload while preserving app health elsewhere.
</rollback>
