## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline icon gaps, status-dot offset, micro separators |
| sm | 8px | Label-to-control spacing, compact metadata rows, stepper chip internals |
| md | 16px | Default card padding blocks, form row spacing, header cluster gaps |
| lg | 24px | Task card padding, questionnaire section padding, header horizontal padding |
| xl | 32px | Gaps between shell regions, questionnaire section separation |
| 2xl | 48px | Training intro / questionnaire top-bottom breathing room |
| 3xl | 64px | Full-page overlay breathing room on Welcome, Interview, Done states |

Non-scale component sizing notes: 40px control height is allowed for header step buttons and Likert option pills; 44px minimum hit target is allowed for the unlabeled time-scale toggle, confidence slider thumb hit area, and questionnaire sliders. These are interactive component dimensions, not spacing tokens, and must not be introduced into the spacing scale.

## Evaluation Shell Contract

### Route layout
- Use a dedicated `/evaluation` route that wraps `DashboardDemoShell`; do not modify `/dashboard-demo` or restyle the inner demo shell globally. Source: `80-CONTEXT.md` D-01.
- Add a persistent evaluation header above the demo shell. Header height: 64px minimum, `px-24 py-16` equivalent spacing, full-width border-bottom using `border-slate-800` to match the existing demo header language.
- Keep the existing dark slate workspace, right rail, viewport toggle, and timeline surfaces visually intact unless this contract explicitly overrides them.
- Primary visual focal point on the main live task-taking screen is the synchronized visualization workspace itself (3D cube, map, and dual timeline). The floating task card is a secondary guidance layer and must not visually dominate or obscure the workspace.

### Header structure
- Use 3 zones in one row:
  1. **Session zone (left):** participant ID, current phase title, elapsed timer.
  2. **Stepper zone (center):** 8-step researcher-controlled phase stepper.
  3. **Researcher zone (right):** condition badge, unlabeled time-scale toggle, previous/next step buttons, start/end session actions, rerun training.
- Visually separate the researcher zone with a `border-l border-slate-700/70` and `pl-16` so controls read as utility chrome rather than participant instruction.
- Participant/session identifiers use Geist Mono at 12px/600 inside low-contrast chips.

### Stepper
- Fixed 8-step order: Welcome → Training → Tasks-A → Questionnaire-A → Tasks-B → Questionnaire-B → Interview → Done. Source: D-12 sequence, interpreted literally as 8 explicit steps despite the "7-phase" wording in `80-CONTEXT.md`.
- Stepper uses 8 compact chips in a single row on desktop; current step = accent fill, completed steps = secondary chip with 1px accent inset line, future steps = muted secondary chip.
- Stepper is clickable only from the researcher zone; participant-facing copy uses the current step title but never shows condition names in the center region.

## Interaction Contract

### Condition visibility + toggle treatment
- Do not show participant-facing text labels `Uniform` or `Adaptive` anywhere in task cards, questionnaires, or main header title. Source: D-04, D-05.
- Keep the time-scale toggle visible to participants but unlabeled. Render it as a compact pill switch in the header with no adjacent text, no tooltip, and no legend.
- Accessibility label may be generic (`Change time-scale view`) but must not expose condition names visually.
- Toggle OFF state = secondary track with neutral thumb; ON state = accent track with white thumb.
- Mode switching is instant. No tweening, cross-fade, ghost motion, or animated repositioning during condition change. Source: D-06.

### Disabled editing states
- All rail tabs remain visible. Source: D-03.
- Participant-locked editing affordances render disabled, non-focusable, and visually muted at 40% opacity: slice creation/deletion/resizing, STKDE parameter controls, warp-source switching, generation actions, and any draft-application actions.
- Each affected panel shows one shared helper strip at top: `Setup locked during evaluation.` in 12px label style with a lock icon; do not repeat helper text on every disabled control.
- Researcher override, if implemented, lives only in the header researcher zone; it must not change tab order for participant mode.

### Researcher visibility
- Researcher-only metadata is allowed in the header right zone only: participant ID, subtle condition badge, step navigation, session control, and training restart.
- Condition badge treatment: 12px label chip with muted slate background, 8px status dot, and compact text only in the researcher zone. Use neutral dot for linear state and accent dot for adaptive state.
- Do not tint the whole page, task card, or questionnaire by condition.

## Training, Tasks, and Forms

### Training tour presentation
- Training phase opens with a centered intro card above a dimmed but visible demo shell (`bg-slate-950/70` scrim). Card max-width: 640px.
- Intro card contains a 5-item checklist in this order: rotate 3D cube, brush timeline, read slice labels, interpret density color, try the unlabeled time-scale toggle. Source: D-13.
- Primary training CTA label: `Start training tour`.
- Reuse `driver.js` popovers but restyle them to the same dark card language: secondary surface, 16px heading, 14px body, 16px padding, 8px corner radius, no bright default theme.
- Tour uses exactly 5 evaluation steps and ends with a success state plus `Begin tasks` CTA for the researcher.

### Task card behavior
- Task card is a non-blocking floating overlay anchored top-left of the viewport area, below the header, with a 24px inset from header and left edge. Source: D-11.
- Card width: 360px fixed on desktop; height auto; background secondary with 92–95% opacity and backdrop blur.
- Card contents order:
  1. Current task label (`Task 1 of 4`)
  2. Task title
  3. Task instruction body
  4. Researcher response controls
  5. Confidence slider shown only after answer capture
- Task order inside each condition is fixed: `T4 → T1 → T2 → T3`. Source: D-10.
- After the researcher records the participant answer, replace the footer with a required 1–5 confidence slider and `Record confidence` CTA before advancing.
- Task card never covers the right rail or timeline centerline; if viewport shrinks below comfortable width, preserve card position and reduce width to 320px before moving it.

### Questionnaire layout
- Show questionnaires after each condition block only. Source: D-08.
- Freeze task UI and open a centered questionnaire shell above a dimmed workspace; max width 960px, max height 80vh, internal scroll area.
- Layout is single-column, sectioned, and completion-safe:
  1. NASA-RTLX section first: 6 stacked question cards, one for each NASA-RTLX dimension, each with statement, discrete 1–20 slider, and left/right endpoint labels.
  2. Interpretability section second: 6 stacked statement rows, each with 5 equal-width option pills labeled 1–5 plus verbal anchors at ends.
  3. Optional open comment textarea last.
- Do not use a dense matrix table for Likert input; stacked rows are required for readability and reduced response error.
- Disable submission until every NASA and Likert item has a value.
- Questionnaire primary CTA label: `Submit condition survey`.

- Post-training CTA: `Begin tasks`

- Confidence CTA: `Record confidence`
- Questionnaire CTA: `Submit condition survey`
- Interview CTA: `Finish interview`
